# Build neoma

Create a production-quality, completely open-source, local-first web application called **neoma**.

neoma is a lightweight Markdown note-taking, research-journal and personal knowledge-management application. It should provide the essential standards users expect from applications such as Obsidian—Markdown files, wiki links, backlinks, tags, frontmatter, daily notes, search, keyboard shortcuts and complete data ownership—without copying Obsidian’s branding, proprietary code or exact interface.

The application must work:

* As a normal web application
* As an installable Progressive Web App
* Completely offline after installation
* Locally through `npm run dev`
* Locally through a production build
* Through Docker for optional self-hosting
* Without requiring an account, cloud service or internet connection
* Without telemetry, analytics, advertisements or tracking

The finished project must be a real, functional application—not merely a landing page, visual prototype or collection of static components.

## Product identity

Product name: **neoma**

Tagline: **Your knowledge, rooted locally.**

Product description:

> neoma is a lightweight, open-source research journal and linked-note application. Your notes remain local, portable and readable as ordinary Markdown files.

The product should feel:

* Fast
* Calm
* Technical
* Academic
* Private
* Open
* Dependable
* Modern without appearing corporate

Create an original neoma logo. Use a simple geometric leaf combined with a document, page or branching-link motif. It must remain recognisable at favicon size.

Do not use Obsidian’s purple branding, crystal imagery, logo, wording or exact interface.

## Design direction

Take visual inspiration from **Modrinth**, especially its:

* Clean dark interface
* Compact navigation
* Rounded panels
* Clear spacing
* Restrained use of colour
* Strong typography
* Subtle borders
* Efficient information density
* Polished hover and focus states
* Practical developer-focused aesthetic

Do not reproduce Modrinth’s exact layouts, logo, icons, components or branding.

Create an original neoma design system.

### Visual system

Use a dark theme by default with an optional light theme.

Dark-theme palette:

* Main background: near-black charcoal
* Sidebar: slightly lighter charcoal
* Raised panels: soft dark grey
* Primary accent: fresh leaf green
* Secondary accent: muted emerald
* Primary text: warm off-white
* Secondary text: neutral grey
* Borders: low-contrast grey
* Warning: amber
* Error: red
* Success: green

Use CSS variables for every design token so themes can be created easily.

Use:

* Rounded corners between 6px and 10px
* Subtle borders rather than heavy shadows
* Clear keyboard-focus outlines
* Smooth transitions between 100ms and 180ms
* Compact but readable spacing
* System fonts by default
* Optional bundled open-source font with no external font request

Do not load fonts, analytics, scripts, icons or assets from external CDNs.

## Core note system

Markdown files must be the primary, canonical note format.

A note should remain usable outside neoma in:

* A plain text editor
* VS Code
* Obsidian
* Logseq
* Joplin after import
* GitHub
* Other Markdown-compatible applications

Do not create a proprietary note format.

Support YAML frontmatter such as:

```yaml
---
title: Transformer Experiment 01
created: 2026-07-20
modified: 2026-07-20
type: experiment
status: active
tags:
  - transformers
  - experiments
aliases:
  - Attention Experiment
---
```

Preserve unsupported frontmatter fields instead of deleting them.

## Storage architecture

Create a storage abstraction so the application does not depend on one browser-storage mechanism.

Implement these adapters:

### 1. Local folder adapter

Where browser support allows it, let the user choose a local folder as their vault.

Read and write ordinary:

* `.md` files
* Image files
* PDF attachments
* Other permitted attachment files

Request file-system permissions only after an explicit user action.

Never upload local files.

### 2. Browser vault adapter

Provide a browser-managed vault using IndexedDB or the Origin Private File System.

This mode must:

* Work offline
* Autosave locally
* Support multiple vaults
* Export a complete vault
* Import a complete vault
* Survive normal page reloads
* Clearly explain that clearing browser-site data may remove browser-managed vaults

### 3. Portable import and export

Support:

* Importing a folder when supported
* Importing individual Markdown files
* Importing a ZIP archive
* Exporting the complete vault as ZIP
* Exporting one note as Markdown
* Exporting one note as HTML
* Exporting one note as PDF through a print-friendly view

ZIP exports must preserve the original folder hierarchy and attachment paths.

Do not make a remote database the source of truth.

## Links and backlinks

For every note, display:

* Incoming links
* Outgoing links

* Orphaned-note status

## Search

Implement fast local full-text search.

Search should support:

* Note titles
* Note contents
* File paths
* Tags
* Properties
* Exact phrases
* Excluded terms
* Filter by folder
* Filter by note type
* Filter by created date
* Filter by modified date

Search must remain entirely local.

Display matching text with highlighted context.

Build the index incrementally so large vaults do not freeze the interface.

Where appropriate, perform indexing in a Web Worker.

## Privacy and security

neoma must be private by default.

Requirements:

* No user accounts
* No analytics
* No telemetry
* No advertisements
* No tracking pixels
* No external API calls
* No note uploads
* No hidden network requests
* No remote AI functionality
* No cloud dependency
* No automatic data collection

Add a privacy screen stating:

> neoma does not collect, transmit or sell your notes or usage data. Your vault remains on your device unless you deliberately export or synchronise it using another tool.

Add a small network-status indicator showing:

* Local
* Offline
* Update available

Do not show an alarming error when offline. Offline use is a normal supported state.

## Performance standards

neoma must be lightweight.

Targets:

* Fast first load
* Lazy-load graph and secondary features
* Avoid large UI frameworks
* Avoid unnecessary dependencies
* Avoid background polling
* Avoid rerendering the entire vault after an edit
* Debounce indexing and autosave
* Virtualise long file and search-result lists
* Use Web Workers for expensive indexing where useful
* Code-split secondary routes and panels
* Keep the initial JavaScript bundle as small as reasonably possible

Display a usable editor quickly rather than waiting for the entire vault index to finish.

## Data integrity

Protect the user’s work.

Implement:

* Debounced autosave
* Visible save state
* Atomic writes where possible
* Recovery copy before destructive operations
* Recently deleted notes
* Confirmation before permanent deletion
* Exportable backups
* Error handling for revoked folder permissions
* Clear warnings before replacing an imported vault
* Conflict detection when a local file changes outside neoma

Never silently discard unsaved text.

## Git compatibility

neoma does not need to implement Git itself.

Make vaults naturally Git-friendly:

* Plain Markdown
* Stable frontmatter formatting
* Relative attachment paths
* No binary database required inside folder-based vaults
* No rewriting unchanged notes
* No random IDs inserted into note content
* Predictable filenames

Include documentation explaining how users can optionally initialise their vault as a private Git repository.

* Daily-note templates function
